Secure Static Website Hosting with S3, CloudFront, ACM,
and Automated w/ Terraform

Built and deployed a secure, production-ready static website using Amazon S3 as the origin, Amazon CloudFront for global content delivery, AWS Certificate Manager for HTTPS, a CI/CD pipeline via GitHub Actions, and Terraform to provision and manage all infrastructure as a reusable template.

Overview

Problem It Solves

Small business and portfolio websites need a hosting solution that is low cost, highly available, secure, and easy to update. This project solves that by using S3 for static storage, CloudFront for caching and global delivery, ACM for HTTPS encryption, and a CI/CD pipeline to automatically publish new content without manual uploads — along with Terraform to easily provision and manage all infrastructure.

What This Project Sets Out to Do

Architecture

How It All Fits Together

Architecture Flow

User → DNS → CDN → Origin → CI/CD → Source Control

Traffic flows from the user through Route 53, into CloudFront (with ACM-managed HTTPS), down to the private S3 origin. On the deployment side, a developer push triggers GitHub Actions which syncs files to S3 and invalidates the CloudFront cache.

When a user visits the website domain, traffic is routed through Route 53 to a CloudFront distribution. CloudFront uses an ACM-managed SSL certificate so the site is always delivered over HTTPS. The CloudFront origin points to a private S3 bucket storing the static files (HTML, CSS, JS). When changes are pushed to the GitHub repository, the CI/CD pipeline uploads updated content to S3 and invalidates the CloudFront cache so users receive the latest version immediately.

Security

Security Considerations

This solution was secured by using HTTPS through ACM and CloudFront, which encrypts all traffic between the user and the edge location. IAM permissions were scoped to least-privilege to limit deployment access. The website content is stored in a private S3 bucket — the bucket is not publicly accessible and can only be read by the CloudFront distribution via an Origin Access Control (OAC) policy. This avoids exposing infrastructure like EC2 and ensures the origin is never directly reachable by end users.

Automation

CI/CD Process

The CI/CD pipeline is triggered when changes are pushed to the main branch in GitHub. The pipeline checks out the code, uploads the updated files to the S3 bucket, and invalidates the CloudFront cache so new content is served immediately. This removes the need for manual uploads and creates a repeatable, reliable deployment process.

Troubleshooting

Challenges & Solutions

Issue HTTPS not working on the custom domain
Cause ACM certificate was not attached correctly to the CloudFront distribution, or the certificate was requested in the wrong region (must be us-east-1 for CloudFront)
Fix Requested and validated the certificate in us-east-1 via ACM, then associated it with the CloudFront distribution
Lesson CloudFront certificate setup has specific regional requirements — ACM certificates for CloudFront must always be in us-east-1 regardless of where the rest of your infrastructure lives
Issue Website changes not showing after deployment
Cause CloudFront was caching the old content at edge locations and continued serving stale files even after S3 was updated
Fix Added a CloudFront cache invalidation step to the GitHub Actions pipeline (aws cloudfront create-invalidation) to run after every deployment
Lesson CDNs greatly improve performance but require active cache management as part of your deployment process
Issue Access denied errors when CloudFront tried to read from S3
Cause The S3 bucket policy was not configured to allow the CloudFront Origin Access Control (OAC) to read objects
Fix Updated the S3 bucket policy to grant s3:GetObject to the CloudFront service principal, scoped to the specific distribution ARN
Lesson Security and access paths must be designed together — the OAC + bucket policy pattern is the modern, recommended way to lock down S3 origins
